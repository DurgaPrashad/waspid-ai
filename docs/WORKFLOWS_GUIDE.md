# Waspid Workflows Guide

Workflows are autonomous workforce executions: the agents generated by
the [Agent Factory](AGENT_FACTORY.md) run as a coordinated team — each
agent executes in its own sandbox, hands its results to the next agents
automatically, and a supervisor step aggregates everything into a final
deliverable.

## Running a workforce

1. Open **Workforce Builder** (`/workforce`), describe an objective,
   and generate a workforce (or load a saved blueprint).
2. Click **Deploy & run workforce**.
3. Watch it execute on **Workflows** (`/workflows`): per-agent status,
   links into each live conversation, the persisted event timeline, and
   the final report when the run completes.

## Execution semantics

- **Entry agents** (no incoming workflow edges) start immediately.
- **Hand-offs** fire automatically when an agent's run terminates: its
  final message is stored in shared workforce memory and injected into
  downstream agents' kickoff prompts.
- **AND-joins** — an agent with several incoming edges waits for all of
  them.
- **Approvals** — an edge marked `requires_approval` holds the
  downstream agent in *Awaiting approval* until a human approves it on
  the Workflows page.
- **Retries** — a failed agent retries automatically (2 attempts by
  default); after that the run fails, and a manual *Retry* on the
  failed task revives it. A 60-minute timeout fails stuck agents.
- **Pause / resume / cancel** — pausing stops new hand-offs (running
  agents finish and their results are recorded); cancelling also
  cancels all queued agents.
- **Aggregation** — after all agents complete, the coordinator (or a
  synthetic Supervisor) runs once more with every output and produces
  the run's final deliverable.
- **Cycles** — cyclic edges are broken deterministically at deploy time
  (recorded in the run's start event); workflows do not loop.

## State and events

Everything is persisted (tables `workflow_run`, `workflow_task`,
`workflow_run_event`): run status and shared context, per-task status /
attempts / conversation ids / outputs, and the full event log
(`workflow_started`, `agent_started`, `agent_completed`,
`agent_failed`, `agent_retrying`, `agent_timeout`, `approval_required`,
`approval_granted`, `workflow_paused`, `workflow_resumed`,
`workflow_completed`, `workflow_failed`, `workflow_cancelled`).

## API

Under `/api/v1/workforce` (Swagger at `/docs`):

| Endpoint | Purpose |
| --- | --- |
| `POST /runs` | Deploy a workforce (blueprint id or inline definition) |
| `GET /runs` | List runs |
| `GET /runs/{id}` | Run detail: tasks + event log + shared context |
| `POST /runs/{id}/pause` / `resume` / `cancel` | Run controls |
| `POST /runs/{id}/tasks/{task_id}/approve` | Approve a held hand-off |
| `POST /runs/{id}/tasks/{task_id}/retry` | Requeue a failed agent |

## Other automation building blocks

- **Event callbacks / webhooks** — subscribe external systems to
  conversation events (see [API.md](API.md)).
- **Chat-ops triggers (enterprise)** — Slack, Jira, and Linear can
  start individual agent runs from external events.
